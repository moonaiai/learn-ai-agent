# Learn AI Agent 写作风格

## 总体口吻

- 使用中文，口吻客观、克制、工程化。
- 不使用“学习笔记”作为标题或章节标签。
- 不写空泛赞美，优先解释问题、原则、工程含义和边界。
- 面向工程读者，默认读者关心可落地性、可验证性和维护成本。
- README 面向学习者，不写成仓库维护手册；配置、发布、workflow、skill 内部规则不要放进 README 正文。

## 标题与结构

- H1 使用主题名或文档名，例如 `12-Factor Agents 设计原则`。
- H2 主章节使用阿拉伯数字编号，例如 `## 1. 背景：Agent 仍然是软件`。
- 术语解释章节命名为 `## 名词解释`，表头固定为：

```markdown
| 名词 | 解释 | 简单例子 |
|---|---|---|
```

- 原文中有固定命名的主题可保留英文，例如 `## Factor 1：Natural Language to Tool Calls`。
- 长文档优先先给“阅读目标”或“核心结论”，再进入背景和细节。

## README 同步规则

每次新增或实质更新 `docs/` 下的文档后，都要检查并按需同步根目录 `README.md`：

- `当前内容`：新增或更新文档入口、主题说明和核心内容。
- `学习路径`：将对应 AI Agent 学习阶段从 `待沉淀` 改为 `已沉淀`，并链接到文档。
- `后续计划`：如果某个计划已经被新文档覆盖，要删除、收窄或改写。
- 不要在 README 中加入 GitHub Pages、workflow、Jekyll、写作 skill 等维护配置说明。

README 的学习路径应面向完整 AI Agent 能力路线，而不是只围绕当前已有文档。已有文档是路线图里的已沉淀节点，未来文档沉淀后再回填对应阶段。

## 段落风格

- 每段只表达一个核心意思，避免堆叠多个抽象判断。
- 优先使用“问题 -> 原则 -> 工程含义 -> 边界/误区”的解释顺序。
- 解释概念时给简单例子，避免只给定义。
- 涉及方案取舍时，用表格列出维度、差异和适用场景。

## 表格风格

常用表格类型：

- `名词 / 解释 / 简单例子`
- `维度 / 检查项 / 期望状态`
- `序号 / 原则 / 面向的问题 / 设计取向`
- `层次 / 相关原则 / 作用`
- `方面 / 框架优先 / 12-Factor 取向`

表格内容要短句化，避免把整段长解释塞进单元格。

## 图片规范

- 正文图片必须本地化保存，不使用远程图片 URL。
- 图片放在主题目录下的 `figures/` 或既有图片目录中。
- Markdown 使用相对路径，例如：

```markdown
![010-software-dag](figures/010-software-dag.png)
```

- 图片附近应有文字解释其作用，不让图片孤立出现。

## 代码与示例

- 代码块只用于解释结构或关键流程，不追求可直接运行。
- 示例应短小，突出概念，例如 tool call、事件线程、控制流伪代码。
- JSON、Python、text 等代码块要标明语言。

## 两类现有文档风格

### 原则型文档

代表文档：`docs/12-factor-agents/12-factor-agents-principles.md`

写法：

- 开头交代资料来源和阅读目标。
- 先给设计背景，再说明核心问题。
- 对每个原则按“问题、原则、工程含义、边界或误区”拆解。
- 使用生产落地检查表和实现骨架收束。

### 论文/资料型文档

代表文档：`docs/context-engineering-2.0-pdf/context_engineering_2_cn_notes.md`

写法：

- 开头给论文链接和核心结论。
- 名词解释靠前，帮助读者进入语境。
- 按论文主线编号展开。
- 图表后解释其论证作用。
- 最后总结工程启发，而不是只复述论文内容。

## 禁止事项

- 不使用 `【学习笔记】`、`学习笔记` 作为文档风格标记。
- 不把远程图片链接直接放进正文。
- 不写“本文将深入探讨”等模板化开场。
- 不堆砌抽象标签，例如“智能化、平台化、体系化”，除非后文有具体解释。
- 不在无依据时夸大效果或给出确定性结论。
