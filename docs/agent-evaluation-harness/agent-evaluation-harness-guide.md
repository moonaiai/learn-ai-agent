# Agent Evaluation Harness：从感觉评估到可复现评估

参考来源：[Agent Evaluation Harness 完全指南 —— 让你的 Agent 评估不再玄学](https://github.com/adongwanai/AgentGuide/blob/main/docs/02-tech-stack/26-agent-evaluation-harness-guide.md)

## 阅读目标

Agent Evaluation Harness 不是单个指标，也不是一份 benchmark，而是一套把 Agent 放进任务、环境、工具、评分器和报告流程里的评估基础设施。

这篇文档保留原文中最重要的概念、流程图和工具选型判断，删除大段安装命令和完整代码实现，重点回答四个问题：

1. Evaluation Harness 到底解决什么问题。
2. Agent 评估为什么比传统 LLM 评估更难。
3. 一个可落地的 Harness 应该有哪些组件。
4. 不同评估工具分别适合什么场景。

## 名词解释

| 名词 | 解释 | 简单例子 |
|---|---|---|
| Benchmark | 标准化测试题目或任务集合，回答“考什么”。 | MMLU、GSM8K、GAIA、WebArena |
| Metric | 评分规则或量化指标，回答“怎么打分”。 | accuracy、pass@1、faithfulness、success rate |
| Harness | 组织评估执行的基础设施，回答“怎么稳定地考”。 | 任务加载、环境重置、Agent 执行、轨迹记录、评分、报告 |
| Task | 一条评估任务，包含输入、初始状态、可用工具、期望结果和评分标准。 | “修复代码 bug，并让测试通过” |
| Trial | 同一个任务的一次执行。Agent 非确定性较强，所以通常要多次运行。 | 同一个 Web 操作任务跑 5 次，统计成功率 |
| Transcript | Agent 的完整执行轨迹。 | observation、thought、tool call、tool output、error |
| Grader | 对结果或轨迹进行评分的组件。 | 精确匹配、规则评分、LLM-as-Judge、人工评分 |

## 1. Evaluation Harness 解决的问题

手动评估的问题不是“样本少”这么简单，而是不可复现、不可比较、不可回归。

如果只手动问 Agent 20 个问题，再凭感觉判断“还不错”，上线后很难回答这些问题：

- 新版本比旧版本强在哪里，弱在哪里。
- 同一任务多次运行是否稳定。
- 工具调用是否走了正确路径。
- 失败是模型推理问题、工具问题、环境问题，还是评分问题。
- 这次修 prompt 有没有破坏之前已经修好的 case。

Evaluation Harness 的核心价值是把评估变成可重复的软件流程：

| 环节 | 需要固化的内容 | 产出 |
|---|---|---|
| 任务定义 | 输入、初始状态、可用工具、期望结果、难度、类别 | 可版本化的 eval cases |
| 执行过程 | 多轮交互、工具调用、环境反馈、超时、异常 | 可回放的 transcript |
| 评分方式 | 精确匹配、规则、代码执行、LLM-as-Judge、人工复核 | 多维度 scores |
| 统计分析 | 多次 trial、成功率、方差、置信区间、失败聚类 | 可比较的报告 |
| 工程集成 | CI、阈值、回归检测、定期重跑 | 可持续的质量门禁 |

## 2. Harness、Benchmark、Metric 的关系

这三个概念经常混在一起，但它们处在不同层次。

```text
┌─────────────────────────────────────────────────────┐
│                    Harness（考场）                    │
│  ┌─────────────┐    ┌─────────────┐                 │
│  │ Benchmark   │    │ Metric      │                 │
│  │ （考试大纲） │ -> │ （评分标准） │ -> 最终成绩       │
│  └─────────────┘    └─────────────┘                 │
│                                                     │
│  负责：环境搭建、任务调度、结果收集、报告生成            │
└─────────────────────────────────────────────────────┘
```

更直接地说：

| 概念 | 回答的问题 | 工程含义 |
|---|---|---|
| Benchmark | 要测哪些题。 | 任务来源和覆盖范围。 |
| Metric | 每道题怎么判。 | 评分口径和业务目标。 |
| Harness | 怎么把评估稳定跑起来。 | 调度、环境、执行、记录、报告和回归。 |

因此，下载一份 benchmark 并手动跑一遍，不等于建立了 Harness。Harness 至少要让另一个人在同样配置下复现结果，并能对比不同版本的变化。

## 3. 为什么 Agent 评估更难

传统 LLM 评估通常是单轮输入输出：

```text
输入 -> [Model] -> 输出 -> 打分
```

Agent 评估要评估一个行动过程：

```text
输入 -> [Agent] -> 观察 -> 思考 -> 调工具 -> 环境反馈
                 -> 再思考 -> 再调工具 -> ... -> 最终结果 -> 打分
```

难点主要来自四个方面：

| 难点 | 具体表现 | Harness 需要提供的能力 |
|---|---|---|
| 非确定性 | 同一个任务可能成功一次、失败一次。 | 多次 trial、统计显著性、置信区间。 |
| 多轮轨迹 | 只看最终答案无法解释为什么失败。 | 逐步记录 observation、tool call、tool output、error。 |
| 环境依赖 | Web、数据库、文件系统、代码运行环境都会影响结果。 | 可重置环境、隔离执行、环境版本锁定。 |
| 多维评分 | 正确率高不代表安全、便宜、稳定。 | 正确性、效率、成本、安全、一致性共同评分。 |

这也是 Agent Harness 和普通 prompt 测试工具的关键区别：它要能评估“做事的过程”，而不只是评估“说出来的答案”。

## 4. Agent Evaluation Harness 的核心组件

一个可用的 Harness 通常包含 Task、Trial、Transcript、Grader、Report 五类核心对象。

```text
┌────────────────────────────────────────────────────────────┐
│                     Evaluation Harness                     │
│                                                            │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐                 │
│  │ Task     │  │ Trial    │  │ Grader   │                 │
│  │ （任务） │  │ （试验） │  │ （评分） │                 │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘                 │
│       │             │             │                       │
│       ▼             ▼             ▼                       │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐                 │
│  │Transcript│  │ Outcome  │  │ Report   │                 │
│  │ （轨迹） │  │ （结果） │  │ （报告） │                 │
│  └──────────┘  └──────────┘  └──────────┘                 │
└────────────────────────────────────────────────────────────┘
```

### 4.1 Task：把“要测什么”写清楚

一条好的 Agent 评估任务至少应包含：

| 字段 | 作用 |
|---|---|
| task_id | 方便追踪、复现和回归。 |
| description | 用自然语言描述 Agent 要完成什么。 |
| initial_state | 起始环境，例如文件、数据库、网页状态、输入材料。 |
| available_tools | Agent 本轮允许使用的工具和权限。 |
| expected_outcome | 成功时应该达到的结果或状态。 |
| grading_criteria | 评分维度，例如正确性、效率、安全性、是否引入副作用。 |
| difficulty/category | 后续按难度、任务类型做分层分析。 |

Task 的重点不是写成复杂数据结构，而是让“任务边界”和“成功标准”可以被版本管理。

### 4.2 Trial：用多次执行衡量稳定性

Agent 的结果受模型采样、工具返回、环境状态和上下文变化影响。对一个重要任务只跑一次，结论通常不可靠。

更合理的报告方式是：

| 指标 | 含义 |
|---|---|
| success_rate | 多次 trial 中成功的比例。 |
| consistency | 同一任务多次结果是否稳定。 |
| avg_duration | 平均耗时。 |
| avg_tokens / cost | 平均 token 和成本。 |
| failure_modes | 失败原因聚类。 |

如果一个任务 5 次里成功 3 次，它不是“通过”，而是“60% 成功率，稳定性不足”。这类表达比单次 pass/fail 更接近生产真实风险。

### 4.3 Transcript：不要只保存最终答案

Agent 失败时，最终答案往往只能说明“错了”，不能说明“为什么错”。

Transcript 应记录：

| 记录项 | 用途 |
|---|---|
| observation | Agent 当时看到了什么。 |
| thought / plan | Agent 的计划或推理摘要。 |
| action / tool_name | Agent 决定做什么。 |
| tool_input | 工具调用参数。 |
| tool_output | 外部环境返回什么。 |
| error | 工具、环境或解析是否出错。 |
| timestamp / step | 复盘执行顺序和耗时。 |

有了轨迹，评估结果才能进一步用于调试 prompt、工具 schema、环境初始化、权限控制和模型选择。

### 4.4 Grader：不同任务使用不同评分器

Grader 不应该只有一种。常见评分方式可以按任务类型组合使用：

| 评分器 | 适用场景 | 主要风险 |
|---|---|---|
| 精确匹配 | 答案明确、输出格式固定。 | 对表达变化过于敏感。 |
| 规则评分 | 代码测试、状态检查、结构化字段校验。 | 规则维护成本较高。 |
| 执行评分 | 代码能否运行、SQL 是否返回正确结果、Web 状态是否达成。 | 需要安全隔离和可重置环境。 |
| LLM-as-Judge | 开放式回答、摘要、对话质量、复杂任务结果。 | 位置偏见、长度偏见、模型偏好、评分漂移。 |
| 人工评分 | 高风险或业务语义复杂的 case。 | 成本高，不适合全量自动化。 |

对于生产级 Agent，推荐把评分拆成“硬规则 + 软判断”两层：能确定性验证的部分先用规则，开放式质量再交给 LLM-as-Judge 或人工抽检。

## 5. 通用架构

原文中的通用架构可以保留为一个完整视图。它表达的是：Harness 不是一个 run 函数，而是一条评估流水线。

```text
┌─────────────────────────────────────────────────────────────────┐
│                        Evaluation Harness                       │
│                                                                 │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ Task Manager（任务管理器）                               │   │
│  │ - 加载任务定义                                           │   │
│  │ - 任务分组和筛选                                         │   │
│  │ - 任务依赖管理                                           │   │
│  └──────────────────────┬───────────────────────────────────┘   │
│                         │                                       │
│  ┌──────────────────────▼───────────────────────────────────┐   │
│  │ Environment Manager（环境管理器）                         │   │
│  │ - Docker / sandbox / browser / DB 环境                    │   │
│  │ - 环境重置和清理                                         │   │
│  │ - 生产配置对齐                                           │   │
│  └──────────────────────┬───────────────────────────────────┘   │
│                         │                                       │
│  ┌──────────────────────▼───────────────────────────────────┐   │
│  │ Agent Interface（Agent 接口）                             │   │
│  │ - 统一调用接口                                           │   │
│  │ - 适配不同 Agent 框架                                    │   │
│  │ - 记录工具调用和状态变化                                 │   │
│  └──────────────────────┬───────────────────────────────────┘   │
│                         │                                       │
│  ┌──────────────────────▼───────────────────────────────────┐   │
│  │ Execution Engine（执行引擎）                              │   │
│  │ - 多轮交互执行                                           │   │
│  │ - 并行 / 串行调度                                        │   │
│  │ - 超时和异常处理                                         │   │
│  │ - 轨迹记录                                               │   │
│  └──────────────────────┬───────────────────────────────────┘   │
│                         │                                       │
│  ┌──────────────────────▼───────────────────────────────────┐   │
│  │ Grading System（评分系统）                                │   │
│  │ - 精确匹配评分器                                         │   │
│  │ - 规则评分器                                             │   │
│  │ - LLM-as-Judge                                           │   │
│  │ - Human-in-the-loop                                      │   │
│  └──────────────────────┬───────────────────────────────────┘   │
│                         │                                       │
│  ┌──────────────────────▼───────────────────────────────────┐   │
│  │ Report Generator（报告生成器）                            │   │
│  │ - 统计分析                                               │   │
│  │ - 对比报告                                               │   │
│  │ - JSON / HTML / dashboard 导出                            │   │
│  └──────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
```

工程上可以从最小版本开始，不必一次做完整平台：

| 阶段 | 最小能力 | 适合时机 |
|---|---|---|
| V0 | 固定任务集 + 手动运行 + 保存结果。 | 验证评估口径是否有用。 |
| V1 | 自动执行 + transcript + 多次 trial + 汇总报告。 | Agent 已进入稳定迭代。 |
| V2 | CI 阈值 + 回归检测 + 失败聚类。 | 多人协作或频繁改 prompt / tools。 |
| V3 | 环境池、并行调度、dashboard、人工复核流。 | 评估成本和覆盖规模成为瓶颈。 |

## 6. 工具选型

没有“最好的 Harness”，只有更匹配当前评估目标的工具。

| 工具 | 主要场景 | Agent 评估 | RAG 评估 | 安全扫描 | 适用判断 |
|---|---|---:|---:|---:|---|
| [lm-evaluation-harness](https://github.com/EleutherAI/lm-evaluation-harness) | LLM 基础能力评估 | 弱 | 无 | 无 | 标准 benchmark、模型选型、论文复现。 |
| [OpenCompass](https://github.com/open-compass/opencompass) | 中英文模型评估 | 弱 | 无 | 无 | 中文 benchmark、国产模型、排行榜对比。 |
| [Promptfoo](https://github.com/promptfoo/promptfoo) | Prompt 测试和安全扫描 | 弱 | 基础 | 强 | 快速回归、CI、red teaming。 |
| [DeepEval](https://github.com/confident-ai/deepeval) | LLM 应用测试 | 中 | 中 | 强 | Pytest 风格、应用级指标、CI 集成。 |
| [RAGAs](https://github.com/explodinggradients/ragas) | RAG 质量评估 | 弱 | 强 | 无 | faithfulness、answer relevancy、context precision/recall。 |
| [AgentBench](https://github.com/THUDM/AgentBench) | Agent 综合能力 benchmark | 强 | 无 | 无 | 多环境、多轮交互、学术对比。 |
| [WebArena](https://github.com/web-arena-x/webarena) | Web Agent 评估 | 强 | 无 | 无 | 浏览器操作、真实网站任务。 |
| [GAIA](https://huggingface.co/datasets/gaia-benchmark/GAIA) | 通用 Agent 复杂任务 | 强 | 无 | 无 | 多步推理、工具使用、文件/搜索/计算混合任务。 |

### 6.1 快速决策树

```text
你需要评估什么？
│
├── LLM 基础能力（知识 / 推理 / 代码）
│   ├── 学术论文或标准榜单 -> lm-evaluation-harness
│   ├── 中文模型 -> OpenCompass
│   └── 快速多模型对比 -> Promptfoo
│
├── RAG 系统
│   ├── 检索 + 生成质量 -> RAGAs
│   └── 端到端测试 + 安全 -> DeepEval
│
├── Agent 能力
│   ├── 综合能力 -> AgentBench
│   ├── Web 操作 -> WebArena
│   ├── 通用复杂任务 -> GAIA
│   └── 业务自定义场景 -> 自建 Harness
│
├── Prompt 优化
│   └── Promptfoo
│
└── 安全扫描
    ├── Promptfoo
    └── DeepEval
```

### 6.2 常见组合

| 团队场景 | 推荐组合 | 原因 |
|---|---|---|
| LLM 研发团队 | [lm-evaluation-harness](https://github.com/EleutherAI/lm-evaluation-harness) + [OpenCompass](https://github.com/open-compass/opencompass) | 覆盖标准能力和中文能力。 |
| Agent 研发团队 | [AgentBench](https://github.com/THUDM/AgentBench) + [DeepEval](https://github.com/confident-ai/deepeval) + 自建 Harness | 标准任务对比与业务任务回归并行。 |
| RAG 产品团队 | [RAGAs](https://github.com/explodinggradients/ragas) + [DeepEval](https://github.com/confident-ai/deepeval) + [Promptfoo](https://github.com/promptfoo/promptfoo) | 检索生成质量、应用测试和安全扫描一起看。 |
| 应用开发团队 | [Promptfoo](https://github.com/promptfoo/promptfoo) + [DeepEval](https://github.com/confident-ai/deepeval) | 低成本接入 PR 回归和 prompt 迭代。 |

自建 Harness 的判断标准很简单：如果你的核心成功标准依赖业务环境、工具副作用、长链路轨迹或内部数据，那么通用工具通常只能覆盖一部分，最终仍需要自建任务层和评分层。

## 7. 评估用例设计

评估用例不应该只覆盖“最正常”的 happy path。更合理的结构是分层覆盖：

| 层次 | 目的 | 示例 |
|---|---|---|
| Smoke tests | 快速发现基础能力是否损坏。 | Agent 是否能正常启动、解析工具返回、完成最简单任务。 |
| Core tests | 覆盖产品主要任务。 | 代码修复、信息检索、订单查询、文档问答。 |
| Edge cases | 暴露边界条件。 | 空输入、超长输入、工具失败、权限不足、格式异常。 |
| Regression tests | 防止已修复问题回归。 | 历史线上 bug、投诉 case、安全绕过 case。 |
| Adversarial tests | 检查安全和鲁棒性。 | prompt injection、越权指令、敏感数据请求。 |

评估集来源也应该多样化：

| 来源 | 价值 | 注意事项 |
|---|---|---|
| 人工设计 | 能覆盖关键业务假设。 | 容易过于理想化。 |
| 生产日志采样 | 贴近真实用户行为。 | 需要脱敏、去重、质量筛选。 |
| 线上失败 case | 回归价值高。 | 要保留失败原因和修复版本。 |
| LLM 辅助生成 | 快速扩展覆盖面。 | 需要人工抽检，避免生成虚假或无意义 case。 |

## 8. CI/CD 集成

评估应该进入开发流程，而不是发布前临时跑一下。

| 触发时机 | 跑什么 | 阈值建议 |
|---|---|---|
| 每个 PR | Smoke tests + 核心回归 case。 | 必须通过，或成功率不能低于主干基线。 |
| 合并到 main | 更完整的核心评估集。 | 关键指标下降超过阈值时阻断。 |
| 定时任务 | 全量评估、red team、安全扫描。 | 生成趋势报告，不一定阻断发布。 |
| 模型或 prompt 变更 | 全量相关任务 + 历史失败 case。 | 重点看回归和成本变化。 |

PR 阶段不要追求全量覆盖，重点是快和稳定。全量评估适合定时任务或发布前检查。

## 9. 如何解读评估结果

单个平均分很容易误导。Agent 评估至少要按这些维度看：

| 维度 | 要回答的问题 |
|---|---|
| 整体成功率 | 当前版本能完成多少任务。 |
| 一致性 | 同一任务多次运行是否稳定。 |
| 类别差异 | 哪类任务最弱，是搜索、代码、工具使用还是对话。 |
| 难度差异 | easy 能过但 hard 不过，还是基础任务也不稳定。 |
| 效率 | 完成任务需要多少步、多少时间、多少 token。 |
| 失败模式 | 主要失败在工具选择、参数错误、环境异常、推理错误还是安全拒绝。 |
| 回归变化 | 相比上一个版本，哪些 case 变好或变坏。 |

推荐报告格式不是“总分 82”，而是类似：

```text
overall_success_rate: 78%
critical_regression: 2 cases
low_consistency_tasks: 7 cases
top_failure_modes:
  - wrong_tool_selection: 18
  - tool_argument_error: 11
  - environment_timeout: 5
cost_change_vs_baseline: +12%
release_decision: block
```

这样的报告能直接支持工程决策：是否允许合并、该修哪里、是否需要回滚。

## 10. 常见陷阱

| 陷阱 | 问题 | 更好的做法 |
|---|---|---|
| 只跑一次就下结论 | 无法反映 Agent 非确定性。 | 每个关键任务跑多次，报告成功率和一致性。 |
| 只看最终答案 | 无法定位失败原因。 | 保存 transcript，按步骤复盘。 |
| 评估数据泄露 | 指标虚高。 | 训练、调参、评估数据分离，并管理评估集版本。 |
| 只看准确率 | 忽略成本、延迟、安全和用户体验。 | 多维度评分，并给关键指标设阈值。 |
| 评估环境和生产环境不一致 | 离线结果无法预测线上表现。 | 锁定模型、prompt、工具版本和环境配置。 |
| 评估集长期不更新 | 无法覆盖新需求和新风险。 | 从生产日志、失败 case 和安全测试中持续补充。 |
| 过度依赖 LLM-as-Judge | 评分可能有偏见和漂移。 | 使用多 judge、交换位置、规则校验和人工抽检对齐。 |

## 11. 工程落地检查表

| 维度 | 检查项 | 期望状态 |
|---|---|---|
| 任务 | 每个任务有明确输入、初始状态、可用工具和成功标准。 | 可复现、可版本化。 |
| 环境 | Web、DB、文件、代码执行等环境可重置。 | 不受上一次 trial 污染。 |
| 执行 | 有最大步数、超时、异常捕获和重试边界。 | 失败可控，不无限循环。 |
| 轨迹 | 保存工具调用、工具结果、错误和最终 outcome。 | 可回放、可调试。 |
| 评分 | 确定性规则优先，开放式任务再用 LLM-as-Judge。 | 分数口径稳定。 |
| 统计 | 关键任务多次 trial，并报告一致性。 | 不用单次结果代表整体质量。 |
| 回归 | 历史线上 bug 进入 regression set。 | 修过的问题不反复出现。 |
| CI | PR 跑轻量评估，定时跑全量评估。 | 质量门禁可持续。 |
| 报告 | 输出类别、难度、失败模式、成本和版本对比。 | 能指导下一步修复。 |

## 关键结论

Agent Evaluation Harness 的本质是把“感觉这个 Agent 还行”转换成“这个版本在这些任务、这些环境、这些评分标准下，以这些统计结果表现如何”。

对工程团队来说，最重要的不是一开始选最复杂的工具，而是尽早建立三件事：

1. 可版本化的评估任务集。
2. 可回放的执行轨迹。
3. 可进入 CI 和发布决策的评估报告。

当这三件事建立起来，Agent 迭代才不只是 prompt 调参，而是有质量反馈闭环的软件工程过程。
